# GitHub 使用指南

## 1. GitHub 简介

- **Hub**: 中心、汇聚、集合
- **GitHub**: 全球最大的代码仓库，世界上最大的开源社区

---

## 2. 核心概念

### 2.1 Fork
- **定义**: 复刻（英语：fork，又译作派生、分支）就像是两个用户同时保存了这份仓库。
- **作用**: 将别人的仓库复制到自己的账户下，方便修改和贡献。

### 2.2 Issues
- **定义**: 讨论问题的地方
  - **Open**: 未解决的问题
  - **Closed**: 已解决的问题

### 2.3 Pull Request (PR)
- **定义**: 将更改从一个分支合并的请求
- **作用**: 用于贡献代码，请求项目维护者将你的代码合并到主分支

### 2.4 Code Review
- **作用**: 审查代码的质量、风格和逻辑，确保代码符合项目要求

## 12. 常用操作

### 12.1 Fetch Origin
- **作用**: 从远程仓库同步最新代码到本地

### 12.2 Stash
- **作用**: 暂存改动而不是提交，方便切换分支

### 12.3 Git Push -f
- **危险操作**: 强制推送，可能会覆盖远程仓库的提交记录

### 12.4 Revert
- **作用**: 反向操作，抵消上一次的 commit，避免直接修改历史

### 12.5 Amend
- **作用**: 修正最新的 commit，只能用于最新的一次提交

### 12.6 Tag
- **作用**: 发布里程碑，标记重要版本

## 13. 最佳实践

- **命令行优于桌面应用**: 命令行指令更灵活，避免高危操作
- **更多使用 `revert` 而不是 `reset`**: 保持提交历史清晰
- **谨慎使用 `git push -f`**: 避免覆盖远程仓库的提交历史

##  14. Git Merge 操作

### 1 基本概念
- **主干分支**：一般指 `main` 或 `master` 分支，通常用于保存稳定的代码版本。
- **提交 PR（Pull Request）**：在分支开发完成后，向主干分支提交合并请求。
  - **注意**：主干分支的直接提交（非 PR）应谨慎操作，避免引入未经测试的代码。

### 2 常用的合并命令
- **`git merge`**：将目标分支的更改合并到当前分支。
- **`git merge --squash`**：挤压提交，将多个提交合并为一个，保持提交历史简洁。
- **`git rebase`**：变基操作，将当前分支的提交重新应用到目标分支的最新提交之后。

#### 2.1 Squash 和 Merge
- **Squash 和 Merge**：
  - 将多个提交“挤压”到一个提交中。
  - 生成一个干净的提交链路，避免冗长的提交历史。
- **适用场景**：在合并 feature 分支到 main 分支时，清理提交历史。

#### 2.2 Rebase
- **Rebase**：
  - 将当前分支的提交“变基”到目标分支的最新提交之后。
  - 例如，将 `feature` 分支的提交变基到 `main` 分支的最新提交。
- **注意**：
  - Rebase 会修改提交历史，因此需要使用 `git push -f` 强制推送，在多人协作中使用时需谨慎。
  - 可以理解为：
    1. 将 `main` 分支“合并”进 `feature` 分支。
    2. 将 `feature` 分支的提交重新应用到 `main` 分支之后。
- **优点**：保持提交记录相对干净，提交链路清晰。

### 3 查看分支合并情况
- 在仓库的 **Insights** 菜单中，选择 **Network** 选项：
  - 可以查看分支的合并情况，包括提交历史、分支关系等。

### 4 Cherry-pick
- **`git cherry-pick`**：
  - 选择部分提交进行合并，而不是整个分支。
  - 适用于将某个特定提交应用到当前分支。
- **适用场景**：需要合并某个特定提交，而不影响其他提交。

### 5 分支合并冲突
- **冲突场景**：
  - 当两个分支对同一文件的同一行代码进行了修改时，会发生合并冲突（conflict）。
- **解决方法**：
  - 在合并时 Git 会提示冲突文件。
  - 手动修改冲突文件，选择需要的代码版本。
  - 标记冲突已解决（如通过 IDE 或命令行），然后继续合并。

## 15. 在 IDE 中使用 Git
### 1 配置 `.gitignore`
- **作用**：指定 Git 忽略的文件或文件夹。
- **语法**：
  - `文件名`：忽略指定的文件。
  - `/文件夹名/`：忽略指定文件夹下的所有文件。
  - `/*.扩展名`：忽略当前目录下所有指定扩展名的文件。
  - `!文件名`：反向忽略，即恢复对某个文件的跟踪。

### 2 Checkout
- **`git checkout`**：
  - 将远程分支的代码拉取到本地。
  - 例如：`git checkout origin/feature-branch`。

## 16. Rebase 详解
- **Rebase**：
  - 变基操作，将当前分支的提交重新应用到目标分支的最新提交之后。
  - **理解**：
    1. 将 `main` 分支的更改合并到 `feature` 分支。
    2. 将 `feature` 分支的提交重新应用到 `main` 分支之后。
  - **优点**：提交记录整洁，提交链路清晰。
  - **注意**：变基会改变提交历史，必须使用 `git push -f` 强制推送，因此在多人协作中需谨慎使用。

## 17. 回滚操作
- **`git reset`**：用于回滚到某个提交状态。
- **常用模式**：
  - **`--soft`**：仅回滚提交记录，保留工作区和暂存区的更改。
  - **`--mixed`**（默认）：回滚提交记录，保留工作区的更改，但清除暂存区。
  - **`--hard`**：回滚提交记录，同时清除工作区和暂存区的更改（危险操作）。
  - **`--keep`**：回滚提交记录，同时保留工作区和暂存区的更改，但会丢弃冲突的更改。

## 18. 代码合并请求
- **提交 PR 时**：
  - 未提交到远程的代码可以使用 `git commit --amend` 修改提交。
  - 已提交到远程的代码会拒绝 `git push`，需使用 `git push -f` 强制推送，但需谨慎操作。

## 19. 暂存更改
- **`git stash`**：
  - 用于暂存当前工作区的更改，以便切换分支或清理工作区。
- **操作**：
  - `git stash apply`：应用暂存的更改，但保留暂存记录。
  - `git stash pop`：应用暂存的更改，同时删除暂存记录。
- **部分暂存**：
  - IDE 提供了部分暂存的功能（如“Shelf”），可以暂存部分文件的更改。

## 20. 提交记录查询
- **`git blame`**：
  - 查看文件的每一行代码是由谁在何时修改的。
  - 用于追溯代码的修改历史。

## 21. Compare 功能
- **作用**：
  - 在 GitHub 网页或 IDE 中，可以比较不同分支、提交或标签之间的差异。
- **场景**：
  - 在合并分支前，比较目标分支和当前分支的差异。
  - 在回滚代码前，比较当前提交和目标提交的差异。
